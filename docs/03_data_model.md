# Data Model

## Overview

The data model is divided into two logical layers:

- Planning Engine
- Analytics Layer

The Planning Engine is responsible for applying business rules and generating workload and capacity calculations.

The Analytics Layer will consume those calculations and organize them into a Star Schema optimized for Power BI.

---

# Planning Engine

The Planning Engine is the core calculation layer of the solution.

Its purpose is to transform customer demand into manufacturing workload requirements and calculate available factory capacity.

This layer follows a Snowflake-style structure because it prioritizes business rules and calculation logic over reporting performance.

---

## Input Tables

### raw_masterplan

Contains customer demand information.

Main attributes:

- project_number
- project_name
- combination_id
- need_date

Each record represents a project requirement received from the customer.

---

### raw_roster

Contains workforce allocation information.

Main attributes:

- operator_id
- operator_name
- workcenter_id
- shift_id

This dataset is used to calculate labor capacity.

---

## Master Data

### dim_product_type

Stores available Product Types.

---

### dim_production_stage

Stores available Production Stages.

---

### dim_combination

Represents the combination between:

- Product Type
- Production Stage

This table plays a central role in the model and is shared by multiple business rules.

Example:

```text
PT1 + ST1

PT1 + ST2

PT2 + ST1
```

---

### dim_workcenter

Stores manufacturing Work Centers.

Each Work Center owns its capacity and receives workload requirements generated by the Routing Matrix.

---

### dim_shift

Stores shift definitions and labor parameters.

Attributes include:

- Work Hours
- Breaks
- Efficiency

---

### dim_date

Calendar table used throughout the planning process.

Stores:

- Calendar Date
- Month
- Year
- Working Day Flag
- Holiday Flag

---

## Business Rules

### leadtime_rules

Defines how many days before the customer Need Date manufacturing activities must begin.

Formula:

```text
Internal Start Date
=
Need Date
-
Lead Time Days
```

---

### routing_hours

Defines workload requirements for every combination of:

- Product Type
- Production Stage
- Work Center

Measures:

- Manned Hours
- Installed Hours

This table is the primary workload generation driver.

---

# Workload Engine

## vw_workload_detailed

Calculated dataset generated from:

- Master Plan
- Lead Time Rules
- Routing Matrix

The view contains:

- Project
- Combination
- Work Center
- Need Date
- Internal Start Date
- Required Manned Hours
- Required Installed Hours

This dataset preserves project-level traceability.

---

## vw_monthly_workload

Monthly aggregation of the detailed workload dataset.

Granularity:

```text
Month
+
Project
+
Combination
+
Work Center
```

Measures:

- Required Manned Hours
- Required Installed Hours

This dataset is used for future comparison against available capacity.

---

# Capacity Engine

## vw_monthly_capacity

Calculates monthly available capacity.

Generated from:

- Workforce Allocation
- Shift Structure
- Calendar Information

Granularity:

```text
Month
+
Work Center
```

Measures:

- Headcount
- Manned Capacity
- Installed Capacity

# Analytics Layer

The Analytics Layer will consume the outputs generated by the Planning Engine.

Its purpose is to provide a reporting-friendly structure optimized for Power BI.

The layer will follow a Star Schema design.

Planned components:

### Facts

- fact_workload
- fact_capacity
- fact_capacity_snapshot

### Dimensions

- dim_date
- dim_project
- dim_product_type
- dim_stage
- dim_workcenter

---

# Analytical Granularity

The solution supports two analytical perspectives.

## Factory Level

Used for tactical and strategic planning.

Typical questions:

- Do we have enough capacity to support future demand?
- Which months have capacity shortages?
- How does utilization evolve over time?

---

## Work Center Level

Used to investigate manufacturing constraints.

Typical questions:

- Which Work Centers are overloaded?
- Which Work Centers are underutilized?
- Where are the bottlenecks?

---

# Key Design Decisions

### Separate Calculation and Analytics Layers

Business rules and calculations remain in the Planning Engine.

Reporting and visualization logic remain in the Analytics Layer.

---

### Preserve Project Traceability

The model allows users to identify which projects are responsible for workload consumption.

Typical navigation path:

```text
Factory
→ Product Type
→ Project
→ Production Stage
→ Work Center
```

---

### Independent Capacity Models

Labor and machine capacity are evaluated separately through:

- Manned Capacity
- Installed Capacity

This allows independent bottleneck analysis for people and equipment.
